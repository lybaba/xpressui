XPressUI Product Roadmap
========================

Vision
------

XPressUI is positioned as a headless form SDK for business workflows.

The product should optimize for:
- API integrations
- dynamic frontend form behavior
- local persistence and offline resilience
- reliable submission pipelines
- developer-friendly orchestration

It should not try to become a Gravity Forms clone.
It should become a strong programmable form engine first.


Current State
-------------

Completed foundations already in the codebase:
- schema-driven form mounting via `mountFormUI(...)`
- custom element runtime via `FormUI`
- AJV + `final-form` validation
- API submission support
- built-in providers:
  - `reservation`
  - `payment`
  - `payment-stripe`
- dynamic field support:
  - conditional visibility
  - remote options loading
- local persistence:
  - draft autosave / restore
  - offline submission queue
  - retry with backoff
  - dead-letter queue
  - manual requeue
  - one-shot replay
- standalone local admin API:
  - snapshot inspection
  - queue/dead-letter filtering
  - purge / replay / requeue helpers
- test suite and CI
- rewritten product docs and backend integration guide


Strategic Objectives
--------------------

1. Stabilize the public headless schema
2. Formalize providers as a real extension system
3. Strengthen local and offline storage for production use
4. Improve observability and developer operations
5. Add higher-value integration providers
6. Separate engine concerns cleanly from UI concerns


Roadmap Phases
--------------

Phase 1: Core Headless Platform
Status: In progress

Goal:
Turn the current runtime into a stable, versioned, extensible SDK core.

Completed:
- local draft persistence
- offline queue
- dead-letter queue
- local admin/debug API
- provider-based submission payloads

Remaining:
- introduce a versioned public form schema
- add schema validation for the full public config
- add migration utilities between schema versions
- separate provider execution logic from `FormUI`
- standardize runtime event names and payload contracts

Exit criteria:
- a form definition can be serialized, validated, migrated, and executed
- the same schema works across runtime and admin/debug tools


Phase 2: Provider System and Business Integrations
Status: Partially started

Goal:
Make integrations first-class, composable, and predictable.

Completed:
- `reservation`
- `payment`
- `payment-stripe`

Next provider targets:
- `webhook`
- `booking-availability`
- `email`
- `crm`

Required platform work:
- provider registry
- provider config schemas
- request payload transformers
- response normalizers
- provider-specific error contracts
- lifecycle hooks:
  - before submit
  - after success
  - after failure

Exit criteria:
- adding a new provider does not require hardcoding business logic into `FormUI`


Phase 3: Storage and Offline Maturity
Status: In progress

Goal:
Make offline behavior robust enough for real business workflows.

Completed:
- localStorage draft adapter
- queue retries with backoff
- dead-letter handling
- replay and requeue controls

Remaining:
- IndexedDB adapter
- queue retention policy / expiration
- batch operations in local admin API
- queue metrics and health snapshots
- optional encrypted local persistence
- deterministic queue replay scheduling

Exit criteria:
- large queues and drafts can be handled reliably without depending on `localStorage`


Phase 4: Rules Engine
Status: Not started

Goal:
Move from simple conditional visibility to a real business rules engine.

Scope:
- multiple conditions (`AND` / `OR`)
- comparison operators:
  - `equals`
  - `not_equals`
  - `contains`
  - `in`
  - `gt`
  - `lt`
- rule actions:
  - show / hide
  - enable / disable
  - set value
  - clear value
  - fetch options
  - invoke pre-checks

Exit criteria:
- business flows can be expressed as config instead of custom imperative UI glue


Phase 5: Developer Operations and Tooling
Status: In progress

Goal:
Make the SDK easy to inspect, debug, test, and operate.

Completed:
- local admin API
- filtered queue/dead-letter inspection
- backend integration docs

Remaining:
- batch actions:
  - requeue all
  - replay matching entries
  - purge filtered entries
- richer query filters:
  - by age
  - by retry window
  - by error text
- debug event stream helper
- import/export of local queue state
- dedicated diagnostics documentation

Exit criteria:
- support/debug workflows are possible without touching browser storage manually


Architecture Direction
----------------------

The target architecture should keep these layers clean:

1. `schema`
- public types
- config validation
- versioning
- migrations

2. `engine`
- form state
- rules
- validation orchestration
- event emission

3. `providers`
- integration registry
- execution pipeline
- payload/response normalization

4. `storage`
- `LocalStorageAdapter`
- `IndexedDbStorageAdapter`
- future remote adapters

5. `ui`
- `FormUI`
- DOM rendering helpers
- adapter layer only


What Not To Do
--------------

Avoid these for now:
- building a large visual form builder too early
- building a full admin back-office before the SDK core is stable
- coupling provider logic directly into the UI layer
- expanding features faster than the public schema matures


Priority Backlog
----------------

P0:
- version the public form schema
- add schema validation for the full config
- introduce a provider registry abstraction

P1:
- add IndexedDB storage adapter
- add batch admin operations for queue/dead-letter
- add a `webhook` provider

P2:

- add richer diagnostics/telemetry helpers
- add import/export tools for local state


Next Task
---------

The next highest-value task is:

Define and implement a versioned public form schema.

Why this is next:
- the runtime is now feature-rich enough that the config shape needs to be stabilized
- providers, storage, rules, and admin tooling all depend on a durable contract
- without a versioned schema, future changes will become harder to migrate safely

Concrete deliverable:
- add `form.version`
- formalize a public schema object shape
- validate it with AJV
- introduce a migration path for older configs

Suggested implementation order:
- create a new schema module for the public config contract
- define `version: 1`
- add a validator function for incoming form configs
- make `createFormConfig(...)` emit the versioned shape
- make `mountFormUI(...)` and local admin tools consume the validated shape
