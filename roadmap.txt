XPressUI Product Roadmap
========================

Vision
------

XPressUI is now clearly positioned as a headless form SDK for business workflows.

The product should optimize for:
- API integrations
- dynamic form orchestration
- multi-step workflow execution
- resilient local/offline persistence
- developer-friendly observability

It should not become a generic no-code CMS or a Gravity Forms clone.
Its strength is a programmable, integration-first runtime.


Current State
-------------

Major capabilities already implemented in the codebase:
- public hydration/runtime entry via `hydrateFormUI(...)` + `FormRuntime`
- headless runtime via `FormRuntime`
- hydration runtime implementation
- schema validation + migration helpers

- dynamic rules and conditional field runtime
- multi-step / wizard runtime:

  - conditional jumps
  - review steps
  - workflow step targeting
- local persistence:
  - draft autosave / restore
  - offline queue
  - dead-letter queue
  - replay / requeue
  - retention policies
  - IndexedDB hydration + migration
  - storage health checks
- standalone local admin/debug APIs
- file uploads:
  - `multipart/form-data`
  - presigned uploads
  - progress events
  - drag and drop
  - previews / thumbnails
  - camera and document scan flows
  - OCR / MRZ extraction
- render modes and read rendering:
  - `form` / `view` / `hybrid`
  - explicit multi-step activation through `form-multi-step` / `view-multi-step`
  - output renderer registry (`text`, `html`, `image`, `file`, `video`, `link`)
  - media display policies (`thumbnail`, `large`, `link`, `gallery`)
  - output snapshot API + events
- configurable multi-step step UI:
  - progress placement
  - navigation placement
  - back-button behavior
- navigation labels now rely on `navigationLabels` as the canonical contract
- commerce-oriented field primitives:
  - `product-list` (catalog max 20, cart interactions, full-image modal)
  - `image-gallery` (catalog max 20, select/remove, full-image modal)
  - `quiz` (single/multi answer selection, open questions, image-backed answers)
  - floating cart trigger with drawer editor
- catalog and quiz choice limits now support:
  - field-level min/max choice constraints
  - per-item max quantity / max selection limits where applicable
- product demos published via GitHub Pages
- CI / release checks
- structural refactor progress:
  - contracts extracted from the hydration runtime
  - view/output rendering helpers extracted
  - workflow step UI helpers extracted
  - submit/provider event helpers extracted
  - field DOM/event/presentation helpers extracted
  - `src/index.ts` materially reduced and more orchestral
- public type cleanup progress:
  - unused `TFieldConfig` options removed
  - unused `TFormConfig` options removed
  - dead form-builder / preset wiring for removed config attributes cleaned up
  - hydrated public aliases now avoid introducing new `TFormUI*` names into the public surface while keeping compatibility with existing imports
  - public entrypoint tests now prefer hydrated aliases, leaving `TFormUI*` names as compatibility types instead of the recommended naming path
  - runtime event emission now mirrors legacy `form-ui:*` notifications onto `xpressui:*` aliases so new integrations can detach from the old component name without breaking existing listeners
  - test-only mount snippet generation was removed because the package no longer exposes any public standalone mount API
- UI shell preservation progress:
  - standard file uploads now keep a persistent selection shell and only replace a focused dynamic body region
  - QR scan and document scan now keep their upload shell and mutate dedicated result/control regions instead of rebuilding the whole selection area
  - document scan OCR/MRZ insight nodes now also stay stable inside each slot card, so scan previews can update extracted text and MRZ summaries in place instead of recreating those sub-blocks
  - image-gallery and quiz now preserve their catalog and selected-state panels while replacing only the dynamic list content
  - image-gallery and quiz selected-item rows now also stay stable while counts and adjacent selections change, reducing more churn inside already-preserved shells
  - open-answer quiz fields now ship a textarea shell directly in template markup, and the runtime reuses that shell instead of clearing the zone on first render
  - radio-buttons and checkboxes now also ship a card-list shell in template markup, and the runtime updates those option cards in place instead of wiping the whole selection zone
  - QR scan start/live controls and document-scan slot toggles now also stay stable inside their preserved shells, so scan-state updates no longer recreate those button rows
  - product-list inline totals and global cart panel rows now also update in place, so cart quantity changes no longer rebuild the summary/panel subtree after the main catalog shell cleanup
  - view/hybrid output fields now keep a stable `data-view-field` shell with a dedicated body node, so output refreshes no longer recreate the whole rendered view container
  - standard upload file rows now also update in place inside the preserved selection body, so adding/removing files no longer rebuilds the whole file list subtree
  - product-list now keeps card/control shells stable and updates cart state in place while retaining the global cart drawer contract
  - product-list action rows and image-gallery toggle rows now also update in place, so quantity/selection changes no longer clear and rebuild those secondary control containers
  - document-scan preview frames now keep a stable preview shell and only swap image/placeholder state instead of resetting the whole preview subtree
  - field validation feedback now writes plain text into stable error nodes instead of injecting raw HTML strings
  - dynamic remote select fields now reconcile their `<option>` nodes in place, preserving matching options instead of clearing and rebuilding the whole `<select>`

Current built-in provider families:
- booking / reservation
- payment
- approval
- email / crm / webhook
- identity verification


Strategic Objectives
--------------------

1. Stabilize the provider response contract
2. Strengthen remote save/resume and workflow continuation
3. Improve multi-step workflow expressiveness
4. Harden binary/file/document flows for production
5. Improve bundle discipline and API surface clarity
6. Keep the hydration layer as a thin UI adapter over runtime services
7. Allow the hydration layer to operate as a read-oriented `ViewUI` for resource/data rendering


Current Assessment
------------------

The product has crossed the point where broad new capability is the main need.
The core runtime surface is now wide enough for real workflow use cases:
- schema-driven mounting
- headless + hydration runtimes
- workflow rules and multi-step flows
- persistence and resume primitives
- binary/document capture flows
- view/hybrid rendering
- commerce-oriented and interactive field primitives including `quiz`

What remains is mainly product hardening and operational coherence rather than
feature breadth:
- provider contracts still need one clearly enforced narrow result model
- save/resume is technically strong but not fully productized for cross-device use
- file/document flows need stronger backend and privacy semantics for production
- admin/debug building blocks exist, but operator workflows are still thin
- docs/demos need to mirror the canonical backend and operational scenarios exactly
- hydrated demos should embed the HTML shell they need and prefer `hydrateFormUI(...)` so showcase pages exercise the same ownership model as `iakpress-console`

Planning principle for the next lots:
- do not add many new primitives before provider/save-resume/file contracts are tighter
- prioritize leverage over breadth
- finish contract-heavy runtime work before expanding showcase/demo surface
- treat backend/console-rendered HTML as the preferred presentation source whenever a hydration contract exists
- keep demos aligned with that rule by shipping inline field shells in `demos/*.html` and using hydration as the default demo contract
- keep the main package entry focused on runtime + hydration only


Demo Hydration Status
---------------------

Current direction:
- `iakpress-console` owns the canonical field HTML
- `xpressui` should hydrate that HTML instead of generating it at runtime whenever a shell contract exists
- demo pages should mirror that backend-first ownership model so bundle size pressure is visible during local development too

Completed:
- every hydrated demo page under `demos/` now embeds its required form shell directly in HTML
- demo pages now use `hydrateFormUI(...)`
- the main `@lybaba/xpressui` entrypoint now matches that runtime/hydration-focused surface
- hydrated demo pages now import `../dist/xpressui.mjs`, which is now the single public runtime/hydration bundle
- `headless-runtime.html` remains the intentional exception because it demonstrates `FormRuntime` without the hydration layer

Follow-up:
- keep new demos on the same contract by default


Roadmap Phases
--------------

Phase 1: Core Platform Stabilization
Status: Complete

Goal:
Keep the SDK core stable and predictable before adding broader surface area.

Completed:
- versioned public schema
- schema validation helpers
- migration helpers
- provider registry extraction
- engine / dynamic / persistence / upload / steps runtimes
- public headless runtime composition
- stricter public API boundary manifest
- stable vs advanced/internal API documentation

Remaining:
- standardize provider response/result contracts

Exit criteria:
- integrators can rely on a narrow documented contract without reading internals


Phase 2: Provider System Maturity
Status: In progress

Goal:
Make providers predictable and interoperable across all workflows.

Implemented providers:
- `reservation`
- `payment`
- `payment-stripe`
- `webhook`
- `booking-availability`
- `calendar-booking`
- `calendar-cancel`
- `calendar-reschedule`
- `approval-request`
- `approval-decision`
- `approval-comment`
- `email`
- `crm`
- `identity-verification`
- `identity-verification-stripe`
- `identity-verification-webhook`

Remaining:
- richer workflow-aware providers:
  - none in current P0 scope

Completed:
- consistent provider-specific error normalization
- pre-submit / post-success / post-failure lifecycle hooks

Exit criteria:
- new providers can be added with little or no hydration-layer changes
- provider outputs drive workflow transitions consistently


Phase 3: Save, Resume, and Offline Continuation
Status: In progress (cross-device hardening mostly shipped)

Goal:
Make interrupted workflows resumable locally and remotely.

Completed:
- local drafts
- local resume tokens
- remote `resumeEndpoint` groundwork
- full remote save/resume contract:
  - create
  - lookup
  - restore
  - invalidate
- IndexedDB hydration
- retention
- local encryption (best effort)
- deterministic queue replay windows
- queue scheduling diagnostics
- signed token hooks for local/remote resume cache:
  - `signResumeToken(payload)`
  - `verifyResumeToken(payload)`
  - `resumeTokenSignatureVersion`
  - runtime invalid-signature event
- share-code exchange MVP for cross-device continuation:
  - `create-share-code` contract
  - `claim-share-code` contract
  - runtime APIs to claim/restore via share code
- signed token verification events and invalid-signature handling
- share-code abuse controls in runtime:
  - claim throttling
  - attempt caps
  - temporary blocking windows

Remaining:
- backend-enforced signed shareable token exchange policy:
  - server-side rate limits
  - expiration enforcement
  - claim policy parity with client safeguards
- cross-device resume UX
  - better success/error/expired/conflict states
  - clearer share-code creation/claim flows
  - stronger device-to-device continuation affordances

Exit criteria:
- a user can safely leave and resume a workflow later, on the same device or another one


Phase 4: Workflow and Rules Expansion
Status: Mostly complete

Goal:
Move from strong field logic to stronger workflow orchestration.

Completed:
- rule operators:
  - `equals`
  - `not_equals`
  - `contains`
  - `in`
  - `gt`
  - `lt`
  - `exists`
  - `empty`
- rule actions:
  - show / hide
  - enable / disable
  - set-value
  - clear-value
  - fetch-options
  - set-error
  - lock-submit
  - emit-event
- step transition DSL
- compound transition groups
- time/date-based branching
- step review summaries
- workflow state integration
- provider-driven workflow routing policies
- workflow snapshots reusable by admin/debug tools

Remaining:
- broader admin/debug workflows built directly on snapshots

Exit criteria:
- complex business flows can be expressed declaratively without ad hoc glue code


Phase 5: File and Identity Flow Hardening
Status: In progress

Goal:
Turn current file/document support into production-safe business flows.

Completed:
- file upload validation
- preview thumbnails
- drag-and-drop
- presigned upload strategy
- bounded retry policy for presigned upload failures
- retry diagnostics in upload events (`attempt`, `nextRetryAt`, `reason`)
- chunked uploads with resumable local progress markers when configured
- camera capture
- QR scan
- document scan
- OCR / MRZ parsing
- document data redaction controls
- upload policy hooks:
  - virus scan callback
  - content moderation callback
  - file acceptance callback

Remaining:
- backend-compatible resumable binary upload contracts beyond local chunk resume
- stronger OCR / MRZ normalization contracts
- file/document privacy controls by field
  - clearer masking/exclusion behavior in submit/debug/storage paths
  - stronger workflow use of normalized OCR confidence/status fields

Exit criteria:
- file and identity flows are safe enough for real operational workflows


Phase 6: Demo and Developer Experience
Status: In progress

Goal:
Make the product easy to understand, adopt, and validate quickly.

Completed:
- static demo gallery
- code snippets beside demos
- copy-to-clipboard buttons
- collapsible code panels
- dedicated demos for:
  - `view` mode photo gallery
  - `hybrid` ecommerce checkout with product-list and image-gallery
- richer demo coverage:
  - simple one-page approval flow
  - remote save/resume flow
  - provider transition flow
- dedicated “headless only” demo without the hydration layer

Remaining:
- docs that mirror demo scenarios exactly
- richer backend integration examples for:
  - provider envelopes
  - OCR/MRZ normalized document flows
  - remote resume/share-code flows


Phase 7: View Runtime and Resource Rendering
Status: In progress (hardening complete, extending)

Goal:
Let one runtime support both data capture and data presentation so a hydrated
instance can behave as a `ViewUI`.

Completed MVP scope:
- explicit render mode contract:
  - `form`
  - `view`
  - `hybrid`
- output renderer registry by field type:
  - text / html / image / file / document / video / audio / map / link
- media/resource display policies:
  - thumbnail / large / link / gallery / embed
- shared output snapshot API reusable by:
  - debug/admin tools
  - headless consumers
- accessibility baseline for overlays:
  - focus trap for cart/gallery overlays
  - escape-to-close handling in dialog context
  - dialog ARIA semantics (`role="dialog"`, `aria-modal`)
  - focus restoration on close
- secure rich content templates for view blocks:
  - `viewTemplate` / `data-view-template` support for `html` fields
  - escaped placeholder interpolation by default
  - explicit trusted override via `viewTemplateUnsafe`
- richer resource renderers:
  - built-in `audio` renderer
  - built-in `map` renderer (safe embed host allowlist + link fallback)
  - built-in `document` renderer (embedded PDF preview + safe link fallback)
- renderer extension docs + production recipes

Remaining:
- richer resource renderers (advanced embedded docs viewers when needed)


Phase 8: Productization and Operational UX
-----------------------------------------
Status: Not started

Goal:
Turn the stabilized runtime surface into clearer end-user and integrator product flows.

Target scope:
- cross-device resume UX:
  - explicit share-code lifecycle states
  - expiry/conflict/error handling
  - clearer claim/recovery experiences
- document/privacy UX:
  - visible field-level masking/exclusion semantics
  - clearer operator confidence around OCR/MRZ output
- resumable upload UX:
  - backend-compatible upload continuation beyond local chunk markers
  - better interruption/retry states for large uploads
- admin/debug workflows:
  - workflow snapshot inspection
  - replay/debug surfaces for queue, dead-letter, transitions, resume state
- provider integration productization:
  - stronger end-to-end examples
  - richer `nextActions` and message handling patterns

Exit criteria:
- main business flows are not only technically possible, but productized enough for integrators and operators

Constraints:
- avoid coupling runtime to WordPress/Ionic/Redux-specific APIs
- avoid unsafe raw HTML rendering without sanitization controls

Exit criteria:
- the hydration layer can render read-only resource views from existing form config/data
- host apps can override output renderers without patching core runtime


Architecture Direction
----------------------

Target layer ownership:

1. `schema`
- public types
- config validation
- migrations

2. `engine`
- validation
- value normalization
- document data state

3. `workflow`
- rules
- step runtime
- workflow state transitions

4. `providers`
- provider registry
- payload building
- response normalization

5. `storage`
- local / indexed / future remote persistence

6. `ui`
- hydrated form runtime
- DOM rendering only
- demo helpers only


What Not To Do
--------------

Avoid these for now:
- building a large visual form builder too early
- adding many new providers without standardizing their result contract
- turning demos into a parallel product surface with custom behavior not backed by runtime APIs
- adding security complexity faster than the save/resume and provider contracts mature

Next Delivery Lots
------------------

Lot A: Provider Contract V2 Tightening
Priority: Highest

Why first:
- it reduces ambiguity across providers, workflow routing, debug output, and docs
- it prevents future providers/examples from adding incompatible shapes

Scope:
- define one canonical normalized provider result shape
- make transition semantics explicit:
  - workflow transition
  - step transition
  - conflict resolution policy
- standardize provider messages/errors payloads
- strengthen strict contract validation and warning behavior
- update canonical docs/examples to use the same envelope

Exit criteria:
- every built-in provider emits the same normalized result structure
- strict mode catches contract drift early
- workflow routing behavior no longer depends on provider-specific quirks


Lot B: Cross-Device Resume Productization
Priority: High

Why second:
- runtime mechanics exist already, but the product experience is still incomplete
- this is one of the highest-value business capabilities already partially shipped

Scope:
- formalize backend-enforced share-code/token exchange policy:
  - server rate limits
  - expiration semantics
  - claim policy parity with client/runtime safeguards
- improve resume UX states:
  - created
  - claimed
  - expired
  - blocked
  - conflict
  - invalid signature
- make device-to-device continuation flows clearer in demos and docs

Exit criteria:
- a cross-device resume flow can be explained, tested, and debugged as a first-class product path
- server/client policy mismatches are eliminated from the contract


Lot C: Production File/Document Hardening
Priority: High

Why third:
- file and identity flows are the most operationally sensitive parts of the runtime
- privacy semantics need to be clearer before broader enterprise use

Scope:
- backend-compatible resumable upload contract beyond local chunk markers
- stronger OCR/MRZ normalized contract:
  - confidence
  - status
  - normalization consistency
- field-level privacy semantics across:
  - submit
  - storage
  - debug/admin
- clearer masking/exclusion rules for document data

Exit criteria:
- file/document flows are operationally predictable under interruption, retry, and privacy constraints
- operators and integrators can reason about document visibility and normalized status safely


Lot D: Snapshot-Driven Admin And Debug UX
Priority: Medium

Why fourth:
- much of the runtime data needed for this already exists

Scope:
- workflow snapshot inspection UX
- queue/dead-letter/replay inspection surfaces

Exit criteria:
- support and integration teams can understand workflow state without reading internals or replaying raw events manually


Lot E: Canonical Docs And Scenario Mirror

Why fifth:
- docs and demos become much more stable once Lots A-C stop moving core contracts

Scope:

Exit criteria:

Next Best Task
--------------

The most immediately profitable next feature set is:

- keep shrinking runtime-owned HTML so `xpressui` mostly hydrates shells coming from `iakpress-console`

Why:
- the biggest runtime-heavy shells have now been cleaned across uploads, QR/document scan, image-gallery, quiz, and product-list
- that makes the remaining DOM-rebuild paths smaller, easier to reason about, and more extractable
- reducing markup ownership inside the hydration layer directly supports the parallel `iakpress-console` hydration work
- `iakpress-console` now treats backend-rendered HTML as the export/preview source of truth, so the next leverage point is to remove more fallback-only shell creation from `xpressui`

Suggested first scope:
- audit field families that `iakpress-console` already classifies as `hydration-ready` or `interactive shell`, and prefer existing backend nodes before creating fallback runtime shells
  - product-list overlays, badges, and secondary controls still created ad hoc inside card bodies
  - image-gallery / quiz selected-state and helper regions that still create rows/cards when backend shells are absent
  - view/review helpers whose body is stable but whose wrapper is still runtime-owned
- keep moving from "runtime creates shell" to "runtime finds shell, mutates state, and only falls back if the shell is missing"
- after each slice, re-audit which field types are still truly `runtime-heavy` in practice rather than only in fallback mode

Follow-up scope (after current P0 shell cleanup):
- cross-device resume UX hardening
- harden signed shareable token exchange contract (policy/rate limits)
- backend-compatible resumable binary uploads
